\section{Stakeholder Analysis}
\label{section:stakeholder-analysis}

The Prometheus project involves multiple parties that influence or are affected by the system's deployment. Identifying these stakeholders ensures that both the technical AI requirements and the business-end objectives are met.

\begin{itemize}
    \item \textbf{Gym Owners and Facility Managers (Primary Users):} Business operators who require data-driven insights to optimize equipment placement and facility layout. They utilize the web dashboard for spatial analytics and ROI assessments.
    \item \textbf{Gym Members (Secondary Users):} Individual athletes and gym patrons who utilize the facility. They are the primary consumers of the real-time occupancy data and progress tracking features. Their main interest is minimizing workout friction and maximizing time efficiency.
    \item \textbf{Product Owner:} The individual responsible for the strategic vision of the Prometheus ecosystem. They ensure that the system remains compliant with data privacy regulations (PDPA/GDPR \cite{thailand:pdpa}) and that the AI capabilities align with the intended problem domain.
\end{itemize}

\section{User Stories and Use Cases}
\label{section:user-stories-use-cases}

\subsection{User Stories}
The following user stories define the core functional requirements from the perspective of the key stakeholders:

\begin{itemize}
    \item \textbf{Gym Member:} "As a gym member, I want to see real-time machine occupancy via my mobile app so that I can plan my sets without waiting for equipment."
    \item \textbf{Gym Member:} "As a gym member, I want to receive an AI-generated workout routine based on currently available machines so that I can have a continuous, uninterrupted workout session."
    \item \textbf{Gym Owner:} "As a gym owner, I want to view a spatial heatmap of my facility so that I can identify 'dead zones' and relocate underutilized equipment to maximize my floor space value."
    \item \textbf{Gym Owner:} "As a gym owner, I want to access historical utilization reports so that I can make informed decisions about purchasing new equipment based on actual demand."
    \item \textbf{Product Owner:} "As a product owner, I want to ensure all raw video data is processed locally at the edge so that the system maintains member privacy and remains compliant with regional data protection laws."
\end{itemize}

\subsection{Use Cases}

The following use cases detail the interaction between the users and the Prometheus system.

\subsubsection{Use Case 1: View and Update Real-Time Machine Status (Fully-Dressed)}
\textbf{Primary Actor:} Gym Member \\
\textbf{Supporting Actor:} Edge Processing Unit (System Actor) \\
\textbf{Goal:} To allow members to identify available equipment in real-time and provide accurate live updates via the mobile application.

\textbf{Stakeholders and Interests:}
\begin{itemize}
    \item \textbf{Gym Member:} Needs low-latency, accurate updates to avoid "availability uncertainty" and plan their workout efficiently.
    \item \textbf{Gym Owner:} Requires a reliable system that encourages member retention by reducing training friction.
    \item \textbf{Product Owner:} Interested in ensuring data transparency and privacy through real-time anonymization and compliance with data protection standards.
\end{itemize}

\textbf{Preconditions:}
\begin{itemize}
    \item The facility has at least 5 HD IP cameras active and streaming via RTSP to the local edge hardware.
    \item The Computer Vision (CV) model is loaded and initialized on local edge-processing hardware.
    \item The member has the Prometheus iOS application open and connected to the service.
\end{itemize}

\textbf{Main Success Scenario:}
\begin{enumerate}
    \item The Edge Processing Unit receives a live video frame from the gym floor.
    \item The CV system identifies machine occupancy states within the facility.
    \item The system validates the "Occupied" or "Free" state using internal occupancy logic.
    \item The Edge Unit extracts anonymized metadata (coordinates and activity labels) and immediately discards the raw video to protect member privacy.
    \item The extracted metadata is transmitted to the cloud-based data management system.
    \item The member opens the Prometheus iOS application to view the live status of the facility.
    \item The application displays a visual map where equipment icons are color-coded: Green for "Free" and Red for "Occupied".
    \item The system triggers a real-time update to all active iOS clients the moment a machine state changes.
\end{enumerate}

\textbf{Extensions (Alternative Flows):}
\begin{itemize}
    \item \textbf{4a. Occlusion Handling:} If the model momentarily loses visual track of a person, temporal smoothing logic prevents a false "Free" status trigger.
    \item \textbf{7a. Notification Subscription:} If a member has subscribed to a specific exercise, the app sends a push notification the moment the required machine becomes vacant.
    \item \textbf{8a. Privacy Restriction:} Members are provided with machine status indicators only and cannot view live CCTV feeds directly for security and privacy reasons.
\end{itemize}

\subsubsection{Use Case 2: Analyze Facility Utilization Trends (Fully-Dressed)}
\textbf{Primary Actor:} Gym Owner \\
\textbf{Goal:} To identify peak usage times and equipment popularity to optimize business operations and capital expenditure.

\textbf{Stakeholders and Interests:}
\begin{itemize}
    \item \textbf{Gym Owner:} Needs data-backed insights to justify equipment investments (CapEx) and floor plan optimization.
    \item \textbf{Facility Manager:} Seeks to optimize staffing, cleaning schedules, and operational efficiency based on traffic volume.
    \item \textbf{Gym Member:} Indirectly benefits from an improved gym layout and a more frictionless training environment.
\end{itemize}

\textbf{Preconditions:}
\begin{itemize}
    \item Historical utilization data has been tracked and stored in the database over a period of time.
    \item The owner has authorized access to the Prometheus Web Dashboard.
\end{itemize}

\textbf{Main Success Scenario:}
\begin{enumerate}
    \item The Gym Owner logs into the PC-based web dashboard.
    \item The owner selects a date range for utilization analysis (e.g., daily, weekly, or monthly).
    \item The system retrieves the "time-in-use" records for every machine from the historical database.
    \item The dashboard generates visual heatmaps of the facility to identify hotspots and "dead zones".
    \item The dashboard provides reports on the most popular equipment and peak usage times.
    \item The owner analyzes the visual intelligence to identify overcrowded areas or underutilized assets.
    \item The owner makes informed decisions regarding facility layout optimization or future capital expenditures.
\end{enumerate}

\textbf{Extensions (Alternative Flows):}
\begin{itemize}
    \item \textbf{4a. Operational Optimization:} The owner identifies traffic trends to adjust staffing or cleaning schedules based on real-time traffic volume.
    \item \textbf{7a. CapEx De-Risking:} The owner decides to remove underused assets identified in "dead zones" to make space for higher-demand equipment.
\end{itemize}
